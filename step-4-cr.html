Knowledge Base

Code Review Playbook — Sample Categories

The playbook accumulates rules as the Tech Lead approves agent-generated PRs. Each category grows independently based on patterns observed across the project's actual code reviews.

Naming & Standards
  • Class/method names must follow D365FO noun+verb convention
  • Extension classes use _Extension suffix
  • Labels required for all user-facing strings
  • No magic strings in business logic classes
Extensibility Patterns
  • Prefer CoC (Chain of Command) over direct override
  • Use SysDictField for dynamic field access
  • Event handlers over method replacement when possible
  • Don't seal classes unless justified in tech spec
Performance
  • No DB calls inside loops — use select with joins
  • Set-based operations preferred over record-by-record
  • Use crossCompany only when required
  • Avoid firstOnly10 when full result set needed
Security & Data
  • All new tables need at minimum a Viewer security privilege
  • Data entities must declare security level explicitly
  • Never skip record-level security (RLS) without TL sign-off
  • Sensitive fields must use PersonalDataCategory
Continuous Improvement

Lessons Learned — Self-Improving Loop

1

Tech Lead acts on PR comment

Tech Lead approves, rejects, or modifies a comment the agent posted. Their decision is captured in the PR review thread via GitHub MCP.

2

Agent synthesises the pattern

Periodically (or on-demand), the agent reads TL decision history and extracts new rules. It filters out one-off exceptions and keeps systematic patterns.

3

Agent creates a Playbook PR

New rules are formatted and submitted as a PR to the Code Review Playbook repository. The PR description explains the origin PR and the decision rationale.

4

Tech Lead approves + merges

After TL review, the rule is merged into the playbook. From the next PR review onward, the agent automatically enforces the new rule.

Active on next review

This loop means the Code Review Agent's knowledge grows with the project — the more PRs reviewed and the more TL feedback captured, the more precisely it enforces the project's own evolving standards.
