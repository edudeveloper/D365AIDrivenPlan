Operating Modes

Choose how the agent works with you

The consultant selects the interaction model that best suits the scenario — from full autonomy to collaborative review.

🤖

Autonomous Mode

The agent generates the full FDD end-to-end and attaches it to DevOps for review. Best for well-defined requirements with clear scope and established patterns.

🤝

Co-authoring Mode

The consultant reviews topic by topic before the final document is assembled. Best for complex or ambiguous requirements where real-time guidance improves quality.

"The agent actively reduces assumptions. When the consultant cannot answer, it posts a comment on DevOps tagging the responsible area owner. When the owner replies, the consultant restarts and the agent picks up the clarifications from the ticket comments."

Risk Management

Attention Points

Known risks and suggestions identified during implementation — each with a corresponding mitigation strategy.

PointTypeDetail
Standard vs GAP classification Risk Agent may make mistakes. Validation with MS Learn evidence partially mitigates this
Clarifications without response Risk If the owner does not reply on DevOps, the FDD remains blocked. Consider timeout/escalation
Decision traceability Suggestion Each classification and clarification should be logged in the FDD for audit purposes
FDD versioning on CRs Suggestion The new FDD should reference what changed compared to the original FDD (semantic diff)
Classification Detail

Requirement Classification Process

✅ Standard

Handled natively by D365FO. The agent identifies out-of-the-box capabilities through MS Learn lookup and marks the requirement as Standard — no development needed.

⚙️ Configuration

Handled via D365FO setup. The agent generates a complete D365FO configuration guide with step-by-step instructions for the functional consultant.

🔧 Customisation (GAP)

Requires development. The agent documents the GAP specification in full detail — this becomes the input for the TDD Agent in Step 2.
